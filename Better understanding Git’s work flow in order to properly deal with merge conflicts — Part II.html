Better understanding Git’s work flow in order to properly deal with merge conflicts — Part II

Tal Goldfus
Jun 30, 2016

What Are Merge Conflicts And Why They Occur

When we run the merge command Git tries to integrate a given branches into our current working branch. Git is taking changes from another context of your code (Which is a snapshot of a branch ) and combines them with your current working files by comparing the working directory’s snapshot(commit) to the one of the other branch.Git will usually figure out how to integrate both snapshots(commits) and merge both versions of the files to the most current one.

Even when changes have been made to the same file in both snapshots Git will most likely be able to figure it out on its own and merge the files without raising any issues .However in some situations Git won’t succeed in merging the files behind the scenes and will ask us to take part in the merge process.

This usually happens when the two snapshots of our code differ from another in the same lines in that same file, or if one snapshot of our file had that line deleted while the other one has that line modified. In these cases Git can’t know with which snapshot of our code he should proceed and will need our help to decide which one of them is correct. Git will then mark the file that contains those conflicts and will ask ask to repair them.Once Git reaches a conflict it does not automatically create a new merge commit. It will pause the process while you resolve the conflict.

Understanding the merge conflicts

When faced with a merge conflict, the first step is to understand what happened.Did one of your colleagues edit the same file on the same lines as you? Did he delete a file that you modified? Did you both add a file with the same name?

If you want to see in which files the merge conflict occurs aka the unmerged paths at any point after a merge conflict, you can run git status . Anything that has merge conflicts and hasn’t been resolved is listed as unmerged:

$ git status
On branch master
You have unmerged paths.
 (fix conflicts and run “git commit”)
Unmerged paths:
 (use “git add <file>…” to mark resolution)
 both modified: some_file.html 
no changes added to commit (use “git add” and/or “git commit -a”)

Git adds standard conflict-resolution markers to the files that have conflicts, so you can open them manually and resolve those conflicts. Let’s take a look on how to solve the most common case, when two changes affected the same file on the same lines.

After we understood in which file the merge conflict occurred we can look into it a little farther and take a quick look at our conflict using the git diff command .

The git diff command gives us a quick way of showing the differences between the two conflicting versions of the file:

$ git diff
diff --cc some_file.html
index 56df44d,9585db7..0000000
--- some_file.html
+++ some_file.html
@@@ -1,3 -1,3 +1,7 @@@
<<<<<<< HEAD: some_file.html
<div id=”footer”> contact : email.support@github.com</div>
=======
<div id=”footer”>
 please contact us at support@github.com
</div>
>>>>>>> [some_other_branch]

Lets focus on understanding the bottom half of the results ,when looking at the git diff results we can see that Git marked the problematic area’s in the file by enclosing it in “<<<<<<< HEAD” and “>>>>>>> [some/other/branch]”.

<<<<<<< HEAD: some_file.html
<div id=”footer”>contact : email.support@github.com</div>
=======
<div id=”footer”>
 please contact us at support@github.com
</div>
>>>>>>> [some_other_branch]

The code from your current working branch is on top, the code from the branch you are merging in is on the bottom separated by “=======” .

The contents after the first marker originate from your current working branch(HEAD ). After the angle brackets, Git tells us where (from which branch) the changes came from( [some_other_branch]).

Solving the merge conflict

Opening the conflicting file and resolving the conflict manually with your file editor is a perfectly valid option , however using a dedicated merge tool can make this much easier .

In order to see what merge tools are available to you run the following command :

git mergetool — tool-help

If you want to switch your default tool to be one of those or any other tool you might have downloaded simply run :

git config --global <merge tool name>

Once your merge tool is configured you can open it using :

git mergetool

For this blog post i used the FileMerge tool which is Apple’s default tool and comes at no additional costs. It might not be the most elegant tool, but it’s definitely a solid one that does the job of comparing & merging text.

However there are some few other tools that are much more impressive and go way further then just merging conflicts , notable ones are Kaleidoscope and DeltaWalker. Other free alternatives include Diffmerge and P4Merge .

FileMerge

After running the git mergetool command our FileMerge GUI tool will look something like this:

Each gray line indicates where there is a difference between the two files, where the left pane is your local file(highlighted in red) and the right pane is the remote file with someone else’s changes(highlighted in yellow). The third pane, at the bottom of the screen shows the final version of our file, given the changes we’ve decided to keep or toss(highlighted in blue).

Using the arrow keys, or by the action selector at the bottom right of the screen we can indicate which pane’s(snapshot) difference/change we would like to keep.

Its that simple ! All that is left to do is save the merge. To tell Git that we are done with this file, we should quit the merge tool to continue. Behind the scenes, this told Git to execute a “git add” command on the (now formerly) conflicted file. This marks the conflict as solved. Should you decide not to use a merge tool and instead clean up the file in your editor, you’ll have to mark the file as resolved by hand (by executing “git add <filename>”).

In the scenario were we have multiple conflicting files we need to just repeat this process (Deciding the version we want to go with , save it, exit the merge tool) until all conflicted files are solve.

Now that all the conflicts have been solved all we need to do is commit the changes and voalá the scary merge conflict is no more.
